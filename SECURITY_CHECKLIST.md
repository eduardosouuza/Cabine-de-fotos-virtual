                                                                                                                                                                                                                              # Checklist de Segurança — React + Supabase SaaS

Use este checklist ao iniciar ou revisar qualquer projeto com React + Supabase.

---

## Banco de Dados (Supabase)

- [ ] RLS habilitado em **todas** as tabelas (`ALTER TABLE x ENABLE ROW LEVEL SECURITY`)
- [ ] Policies de SELECT, INSERT, UPDATE, DELETE definidas em cada tabela
- [ ] Usuário só acessa seus próprios dados (`auth.uid() = user_id`)
- [ ] `REVOKE UPDATE` em colunas críticas (`plan`, `role`) do role `authenticated`
- [ ] Triggers para limites de negócio (ex: fotos por plano, eventos por plano)
- [ ] `service_role` key **nunca** exposta fora do servidor
- [ ] Operações admin protegidas por RLS com check de `role = 'admin'`

---

## Autenticação

- [ ] Rotas protegidas verificam auth no servidor (RLS), não só no cliente
- [ ] Role admin validado via RLS no banco, não só via `profile.role` no frontend
- [ ] JWT obrigatório em todas as Edge Functions que manipulam dados
- [ ] `user_id` nos metadados vem sempre do JWT, nunca do body da requisição

---

## Frontend

- [ ] `.env` no `.gitignore`
- [ ] Nenhuma `SECRET_KEY`, `SERVICE_ROLE_KEY` ou `WEBHOOK_SECRET` em variáveis `VITE_*`
- [ ] Lógica de negócio crítica (limites, permissões, planos) não confia só em estado local
- [ ] Sem `console.log` com dados sensíveis (userId, tokens, valores de pagamento)
- [ ] Limites de plano validados no servidor (trigger/policy), não só no cliente

---

## Pagamentos (Stripe)

- [ ] Webhook valida assinatura (`stripe-signature`) antes de processar qualquer coisa
- [ ] `plan_type` validado contra whitelist antes de salvar no banco
- [ ] `user_id` no metadata vem do JWT, nunca do body da requisição
- [ ] Stripe Publishable Key (`pk_...`) no frontend — OK
- [ ] Stripe Secret Key (`sk_...`) apenas no servidor — nunca no frontend
- [ ] `STRIPE_WEBHOOK_SECRET` apenas no servidor

---

## Storage (Supabase)

- [ ] Policies no bucket definidas (upload, leitura, deleção)
- [ ] Upload validado contra ownership do recurso via RLS ou trigger
- [ ] Arquivos públicos são intencionalmente públicos (não vazar arquivos privados)

---

## Edge Functions

- [ ] CORS restrito ao domínio do frontend em produção (não `*`)
- [ ] Todas as funções que escrevem dados exigem JWT válido no header
- [ ] Inputs validados antes de usar (tipo, formato, valores permitidos)
- [ ] Sem logs de dados sensíveis (userId, planType, valores financeiros)

---

## Vulnerabilidades Comuns a Testar

| Teste | Esperado |
|-------|----------|
| `UPDATE profiles SET plan = 'premium'` direto no Supabase | ❌ Bloqueado (REVOKE) |
| `UPDATE profiles SET role = 'admin'` direto no Supabase | ❌ Bloqueado (REVOKE) |
| INSERT em `media` com `event_id` de outro usuário | ❌ Bloqueado (RLS) |
| DELETE em evento de outro usuário | ❌ Bloqueado (RLS) |
| Exceder limite de fotos do plano | ❌ Bloqueado (trigger) |
| Chamar Edge Function sem JWT | ❌ Retorna 401 |
| Webhook com `plan_type` inválido | ❌ Ignorado/rejeitado |
| Acessar `/admin` sem role admin | ❌ Redireciona |
